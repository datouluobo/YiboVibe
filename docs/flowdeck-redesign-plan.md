# FlowDeck 重做方案定稿

## 1. 页面定位

`FlowDeck` 重定义为“系统总览页”，仅负责展示当前产品主链路的整体状态。

页面回答 4 个问题：

1. 当前处于什么运行模式。
2. 服务端与同步链路是否可用。
3. 本地核心功能当前是否启用。
4. 当前设备与集群概况如何。

页面不再承担以下职责：

- AI 状态页
- 网关调试页
- 设备编辑页
- 技术术语展示页

## 2. 页面边界

### 2.1 保留展示内容

保留以下 4 类信息：

1. 账户与运行模式
2. 同步状态
3. 本地功能状态
4. 设备概览

### 2.2 明确移除内容

以下内容不再出现在 `FlowDeck`：

- `FlowProbe` 入口提示或状态展示
- AI 连通性、激活节点、延迟、模型等信息
- `NAS Hub 对接中`、`WebSocket 执行中`、`HTTPS 安全信道` 这类旧网关语义
- 没有真实检测来源的绿色状态灯
- 设备名本地假编辑能力

## 3. 信息架构

页面固定为 4 个区块，自上而下排列：

1. 顶部总览条
2. 同步与账户
3. 功能状态
4. 设备概览

## 4. 区块设计

### 4.1 顶部总览条

目标：让用户在 3 秒内看懂当前整体状态。

展示 4 张摘要卡：

1. 运行模式
2. 当前账户
3. 服务端状态
4. 设备状态

建议字段：

- `mode`
- `username`
- `server_url`
- `server_status_code`
- `current_device_name`
- `online_device_count`
- `device_count_total`

建议状态值：

- `LOCAL_OFFLINE`
- `REMOTE_SYNC`
- `SERVER_OK`
- `SERVER_UNCONFIGURED`
- `SERVER_UNREACHABLE`
- `SERVER_PENDING`

显示规则：

- 该区块只展示结论，不展开错误详情。
- 本地模式下，服务端状态显示“未使用”。
- 服务端状态只显示短标签，不显示长段错误文本。

### 4.2 同步与账户

这是页面主卡片，负责解释“为什么能同步”或“为什么不能同步”。

建议展示字段：

- 当前模式
- 用户名
- 服务器地址
- 是否记住密码
- 最近一次检测时间
- 同步状态代码
- 同步状态解释
- 技术详情
- 手动重试按钮

建议状态码体系：

- `SYNC_OK`
- `SYNC_LOCAL_MODE`
- `SYNC_AUTH_REQUIRED`
- `SYNC_SERVER_UNREACHABLE`
- `SYNC_API_ERROR`
- `SYNC_UNKNOWN`

每个状态必须同时包含：

- 面向用户的解释
- 面向排障的技术细节

示例：

- 状态：`SYNC_API_ERROR`
- 说明：服务端已响应，但同步检查未通过
- 细节：`HTTP 400`、`HTTP 404`、返回结构不匹配、凭据缺失

交互规则：

- 保留“立即重试”
- 支持展开/折叠“技术详情”
- 不在本页中修改服务器地址、登录状态或账户信息

### 4.3 功能状态

目标：展示“当前功能有没有启用”，不伪装成实时探针。

建议展示模块：

- `FlowSnap`
- `FlowHint`
- `FlowKeys`
- `FlowSync`
- `FlowDrop`

每个模块展示：

- 模块名
- 简短职责说明
- 当前状态
- 可选补充计数

建议状态值：

- `ENABLED`
- `DISABLED`
- `NOT_CONFIGURED`

可选补充信息示例：

- `FlowSnap`：词条数
- `FlowHint`：已启用词库数
- `FlowKeys`：当前启用映射数
- `FlowSync`：本地模式 / 远端模式
- `FlowDrop`：已启用 / 未启用

约束：

- 不显示“Core 引擎运行中”“E2EE 密钥就绪”“键盘 Hook 激活”这类无真实检测来源的状态。
- 如果没有运行态来源，只显示配置状态。

### 4.4 设备概览

目标：只读展示当前设备与集群状态，不承载管理操作。

建议展示字段：

- 当前设备名
- 当前设备类型
- 总设备数
- 在线设备数
- 设备列表
- 每台设备的在线状态
- 最近活跃时间
- 是否当前设备

每条设备建议字段：

- `id`
- `name`
- `device_type`
- `is_online`
- `is_current_device`
- `last_seen_at`

显示规则：

- 当前设备置顶
- 在线设备优先
- 离线设备弱化显示
- 名称超长时截断，悬浮显示完整值
- 不提供设备名编辑能力

## 5. 错误展示规范

### 5.1 问题定义

现有页面把许多“非成功”统一解释成“连接不上”，误导性很强。

### 5.2 新的错误分层

需要明确区分：

1. 网络错误
2. 认证错误
3. 路由错误
4. 服务端业务错误
5. 本地条件不足

### 5.3 统一展示结构

所有状态异常采用统一结构：

- `code`
- `summary`
- `detail`

示例：

- `SYNC_SERVER_UNREACHABLE`
- 无法连接到服务端
- 请求未到达目标主机，可能是地址错误、端口未开放或网络不可达

- `SYNC_AUTH_REQUIRED`
- 缺少本地凭据，无法自动检查
- 当前未保存必要密码或令牌，请重新登录后重试

- `SYNC_API_ERROR`
- 服务端已响应，但请求未通过
- 例如 `HTTP 400`、`HTTP 404`、返回结构不匹配

规范要求：

- 不允许再把所有失败都写成“连接不上”。
- `400/401/403/404/500/超时` 至少要归类到不同错误类型。
- 本地模式不是失败，应显示为“本地模式”或“未启用远端同步”。

## 6. 文案原则

页面文案改为直接、可判断、可排障的表达。

应删除或避免的表达类型：

- 多端协同互联
- 全局时基锚点
- 云数据块量级
- 核心已握手
- 执行中
- 魔法
- 引擎正在侦测云端库

建议改用：

- 当前模式
- 同步状态
- 服务端地址
- 最近检查时间
- 在线设备
- 本地配置状态
- 错误详情

## 7. 与现有实现的差异

相对当前 `FlowDeck.tsx`，应删除：

- `引擎核心` 里固定为 `ok` 的状态徽章
- `NAS Hub 对接中 / WebSocket 执行中 / HTTPS 安全信道`
- 设备名编辑按钮和本地假编辑逻辑
- 将所有异常泛化成“连接不上/探测失败”的逻辑
- 所有 AI 相关展示残留

应保留但重构：

- `get_vault_sync_status` 的结果展示
- `get_cluster_devices` 的设备列表展示
- `FlowDeck` 作为默认首页的定位

## 8. 开发原则

开发顺序建议分两步：

1. 先改信息结构和错误语义，确保展示正确。
2. 再改视觉布局，统一摘要卡与详情卡样式。

重做后，`FlowDeck` 必须满足：

- 只展示当前产品真实存在的主链路能力
- 不再承载 AI 状态
- 不再使用无法验证的绿色状态灯
- 所有错误均输出“状态代码 + 解释”
- 页面交互保持轻量，只做查看和重试
