# YiboVibe 移动端控制台规划

更新时间：2026-05-15

## 1. 文档目的

本文档用于明确 YiboVibe v2 移动端的产品形态、核心功能、页面结构、消息模型与分阶段实施建议。

本文档重点回答：

- 移动端在 v2 中到底承担什么角色
- 移动端主页面应该如何组织
- 输入区、工具行、对话/终端双视图应如何定义
- 文件、图片、FlowMind、Sessions 等能力应如何落位
- 哪些能力适合先做，哪些应延后

产品边界以 [specs-v2.md](/F:/Download/GitHub/YiboVibe/docs/specs-v2.md)、[architecture-vibe-coding.md](/F:/Download/GitHub/YiboVibe/docs/architecture-vibe-coding.md) 与 [mobile-desktop-mvp-plan-2026-05-14.md](/F:/Download/GitHub/YiboVibe/docs/mobile-desktop-mvp-plan-2026-05-14.md) 为准。

## 2. 移动端定位

YiboVibe v2 中，移动端应被定义为 `Remote Console`，而不是桌面端页面的缩小镜像，也不是一个只能查看状态的陪衬应用。

移动端的核心职责是：

- 远程查看桌面端当前活跃开发上下文
- 在多个 Session 间切换并持续观察
- 向桌面端宿主发送输入、命令和确认动作
- 以低摩擦方式投喂资源
- 在异常、等待确认、长时运行场景下完成远程接管

不建议把移动端定义为：

- 直接替代桌面端执行主业务的运行宿主
- 直接浏览或托管桌面全量文件系统的执行端
- 独立于桌面端存在的完整开发环境

原则仍应保持：

- `Headless Host First`
- `Context Before Text`
- `Desktop Executes, Mobile Orchestrates`

## 3. 总体设计目标

移动端主控制台应满足以下目标：

- 一屏内完成查看、输入、确认与切换
- 支持 `终端模式` 与 `对话模式` 两种视图投影
- 让用户离开电脑后仍能低成本继续操控 Session
- 在手机形态下优先减少误触、跳转和高危操作风险
- 与现有服务端 Signal/Session Hub 和桌面端 Session Host 对齐

## 4. 主页面结构

移动端建议收敛为单页主控制台，辅以侧边栏、底部面板和弹层，而不是多级页面来回切换。

推荐结构如下：

```text
顶部状态栏
  -> 当前设备 / 当前 Session / 连接状态 / 告警入口

主视图区
  -> 对话模式 或 终端模式

底部输入区（两行固定）
  -> 第一行：输入框
  -> 第二行：工具行

辅助层
  -> Sessions 侧边栏
  -> 文件/资源面板
  -> FlowMind 面板
  -> 设置面板
  -> 更多动作面板
```

### 4.1 顶部状态栏

应稳定显示：

- 当前连接的桌面设备名
- 当前 Session 标题或 shell 类型
- 当前连接状态
- 最近活动时间
- 告警入口或未处理提示数

不建议在顶栏堆放过多次级按钮，避免压缩主视区。

### 4.2 主视图区

主视图区只承载一条当前 Session 的主内容流，但支持两种显示方式：

- `终端模式`
- `对话模式`

两种模式应共享同一份底层事件流，不应形成两套彼此独立的数据源。

### 4.3 底部输入区

底部固定为两行：

- 第一行：输入框
- 第二行：工具行

这是移动端最关键的交互中枢，应始终稳定停靠在底部，避免随着内容滚动消失。

## 5. 主视图区模式定义

### 5.1 终端模式

终端模式用于尽可能保留桌面端 Session 输出的原貌。

特点：

- 等宽字体
- 接近日志流或终端滚动体验
- 支持快速复制
- 支持跳转到最新输出
- 支持错误高亮
- 支持暂停自动滚动

适用场景：

- 查看 CLI 实时输出
- 跟进构建、测试、部署、日志
- 处理需要观察细节的 shell 命令

### 5.2 对话模式

对话模式不是新的会话，而是把同一条 Session 事件流重新组织为消息流。

建议渲染规则：

- 用户输入显示为右侧气泡
- 终端输出按时间窗口和语义块聚合成左侧气泡
- 系统事件显示为中间系统卡片
- 每条消息带时间戳
- 区分输入、输出、系统、告警、等待确认

适用场景：

- 在手机上更轻量地阅读长会话
- 回看“我发了什么，桌面端回了什么”
- 快速处理 AI/Agent 风格的会话

### 5.3 两种模式的关系

推荐模型：

- 同一 Session
- 同一事件流
- 两种投影方式

不建议模型：

- 终端模式一套数据
- 对话模式另一套数据

否则状态同步、未读计数、时间戳和跳转逻辑会变复杂。

## 6. 输入区设计

### 6.1 输入框

输入框建议作为主发送入口，承担普通文本、命令、模板结果和自定义确认输入。

建议行为：

- 默认单行，最多自动扩展到 3 行
- 按 Session 保存草稿
- 支持发送前编辑
- 支持长按发送按钮选择不同发送方式

建议发送方式：

- `发送并回车`
- `仅填充不发送`
- `发送原样文本`

如后续支持特殊键，还可扩展：

- `发送 Enter`
- `发送 Ctrl+C`
- `发送 y / n`

### 6.2 输入框附属状态

输入框附近建议显示：

- 当前模式：`终端` / `对话`
- 当前目标 Session
- 是否存在待发送附件
- 是否存在等待确认状态

## 7. 工具行规划

工具行位于输入框下方，作为高频能力入口。

建议首批包含以下按钮：

- `Sessions`
- `文件`
- `FlowMind`
- `图片`
- `模式切换`
- `快捷动作`
- `设置`
- `更多`

推荐顺序：

```text
Sessions | 文件 | FlowMind | 图片 | 模式 | 动作 | 设置 | 更多
```

### 7.1 Sessions

作用：

- 打开 Session 侧边栏
- 查看并切换当前桌面设备下的 Session
- 创建新 Session
- 关闭、停止、恢复 Session

### 7.2 文件

文件能力不应只有一个模糊的“打开文件”，而应明确拆成两类：

- `手机本地文件`
- `桌面资源`

原因：

- 手机本地文件适合直接选择并上传
- 桌面文件不应由手机直接浏览桌面全盘
- 桌面资源应通过桌面代理或资源桥接层读取

首期建议：

- 支持手机本地文件选择
- 支持请求桌面端打开文件选择器
- 支持从桌面“最近资源”或“已索引资源”中选择

不建议首期直接实现：

- 手机远程浏览桌面任意路径
- 手机成为桌面文件系统浏览器

### 7.3 FlowMind

`FlowMind` 应作为移动端输入增强层，而不是完整桌面页面搬运。

建议在移动端实现为底部面板，包含三类能力：

- `命令模板`
- `词库候选`
- `快捷注入`

建议行为：

- 选中一项后先填入输入框
- 长按支持直接发送到当前 Session
- 支持按工具、Shell、桌面开发环境切换词库

首批可配置词库方向：

- `Codex`
- `Claude Code`
- `Cursor`
- `PowerShell`
- `Git`
- `Docker`

与 v2 体系的关系：

- `FlowSnap` 负责模板化与高频命令片段
- `FlowHint` 负责候选增强与补全
- `FlowMind` 负责把二者统一投影为移动端可消费的输入加速器

### 7.4 图片

图片能力建议独立成按钮，不与“文件”完全混合。

首期支持：

- 相册选图
- 拍照
- 图片预览
- 作为资源附件挂载到待发送区

在视图中的表现建议：

- `对话模式`：显示为一条用户附件消息
- `终端模式`：显示为系统资源投喂事件

### 7.5 模式切换

用于在 `终端模式` 与 `对话模式` 之间快速切换。

建议为单一按钮或 segmented control，不建议藏在设置里。

### 7.6 快捷动作

快捷动作用于降低交互式终端处理成本。

建议首批内置：

- `Enter`
- `Ctrl+C`
- `y`
- `n`
- `重试`
- `总结最近输出`
- `复制最近错误`

其中一部分应由系统根据当前 Prompt 状态动态置顶。

### 7.7 设置

建议包含：

- 默认模式
- 字体大小
- 时间戳显示方式
- 自动滚动
- 震动/通知
- 移动网络下资源上传限制

### 7.8 更多

可放入低频能力：

- 清空本地视图缓存
- 查看会话详情
- 查看设备详情
- 查看最近告警
- 断开连接

## 8. Sessions 侧边栏设计

`Sessions` 侧边栏建议作为移动端一级核心结构，而不是一个简单列表弹窗。

建议信息项：

- 设备名
- Session 标题
- Shell 类型
- 当前状态
- 最后输出时间
- 未读计数
- 是否等待输入
- 是否存在错误

建议支持的动作：

- 切换当前 Session
- 新建 Session
- 停止 Session
- 关闭 Session
- 恢复 Session
- 置顶或收藏常用 Session

状态颜色建议与控制台规范保持一致：

- 绿色：运行正常
- 黄色：等待确认或部分异常
- 红色：错误、崩溃、断开
- 灰色：未知或已停止

## 9. 消息与事件模型建议

现有实现已具备 `session:stdin`、`session:output`、`session:update` 等基础能力，但若要做好移动端 `对话模式` 和动作卡，建议在现有链路上补齐更稳定的事件模型。

建议移动端消费的标准事件至少包括：

- `user_input`
- `terminal_output`
- `session_state`
- `prompt_request`
- `resource_event`
- `control_event`
- `system_notice`

### 9.1 user_input

建议字段：

- `session_id`
- `text`
- `sender_device`
- `ts`

用途：

- 对话模式渲染用户气泡
- 输入历史与草稿恢复

### 9.2 terminal_output

建议字段：

- `session_id`
- `text`
- `stream`
- `ts`
- `chunk_id`

其中 `stream` 建议支持：

- `stdout`
- `stderr`
- `system`

用途：

- 终端模式原样展示
- 对话模式按块聚合
- 错误高亮

### 9.3 session_state

建议字段：

- `session_id`
- `state`
- `exit_code`
- `ts`

建议状态至少包括：

- `starting`
- `running`
- `paused`
- `waiting_input`
- `stopped`
- `crashed`

### 9.4 prompt_request

用于承接 spec 中的 `State-Aware Action Bar`。

建议字段：

- `session_id`
- `prompt_kind`
- `prompt_text`
- `suggested_actions`
- `ts`

建议首批识别：

- `[y/n]`
- `Press Enter to continue`
- `Retry?`
- `Enter path:`
- `Select file`

### 9.5 resource_event

用于统一描述图片、文件、文本片段、词库模板等投递动作。

建议字段：

- `session_id`
- `resource_type`
- `resource_name`
- `source`
- `status`
- `ts`

### 9.6 control_event

用于描述高危确认、拒绝、超时、中止等动作。

建议字段：

- `session_id`
- `action`
- `result`
- `ts`

## 10. 文件与资源桥接策略

移动端文件能力应遵循 `优先复用桌面本地读取能力，不把移动端变成直接文件访问宿主` 的原则。

建议拆成三类来源：

### 10.1 手机本地资源

包括：

- 图片
- 文本
- PDF
- 日志文件

适合直接上传或转交服务端暂存。

### 10.2 桌面已知资源

包括：

- 最近打开文件
- 桌面端已索引资源
- 当前项目相关文件

建议由桌面端返回列表，移动端只负责选择。

### 10.3 NAS 或私有资源

此类建议延后接入，通过 `FlowSync`、`FlowDrop` 或 Resource Bridge 统一承接，不建议首期单独发明移动端侧协议。

## 11. 对话体验细化建议

为了让移动端尽量接近聊天应用，同时不失去控制台属性，建议补以下体验：

### 11.1 区分消息类型

至少区分：

- 我发送的输入
- 桌面端输出
- 系统事件
- 告警事件
- 资源事件

### 11.2 时间戳

建议默认显示到分钟，展开细节时可看秒级时间。

### 11.3 输出聚合

不要把每一小段输出都渲染成单独消息。

建议策略：

- 同类输出在短时间窗口内合并
- 检测到明显边界再拆新块
- 错误输出单独成块

### 11.4 未读与跳转

建议支持：

- 未读计数
- 跳到最新
- 跳到最近错误
- 跳到最近等待确认

## 12. 安全与高危操作

移动端存在误触风险，所有高危动作都应显式确认。

至少包括：

- 停止 Session
- 关闭 Session
- 重启桌面宿主
- 清理或覆盖资源

建议策略：

- 重要动作二次确认
- 高危动作显示目标 Session 与设备名
- 保留最小审计记录
- 与桌面端已有确认链对齐

## 14. 分阶段实施建议

### Phase 1：把移动端做成可用遥控器

目标：

- 不追求完整聊天化，先把主控制链跑顺

范围：

- Session 侧边栏
- 顶栏状态区
- 两行底部输入区
- 工具行基础按钮
- 终端模式优化
- 时间戳
- 输入/输出/系统事件区分
- 停止/恢复/关闭确认

完成标准：

- 手机上可以稳定查看 Session、切换 Session、发输入、看输出

### Phase 2：补成聊天式控制台

目标：

- 让手机端更适合长时间查看和低摩擦发令

范围：

- 对话模式
- FlowMind 底部面板
- 输出聚合
- 草稿记忆
- 未读数
- 快捷动作
- Prompt 动作卡

完成标准：

- 同一条 Session 既能按终端看，也能按聊天看

### Phase 3：接入资源桥接与增强能力

目标：

- 把移动端从遥控器升级为完整远程控制台

范围：

- 手机文件与图片投喂
- 桌面资源选择
- FlowSync / FlowDrop 接入
- NAS 资源桥接
- 会话摘要
- 告警与自愈入口

完成标准：

- 用户离开电脑后仍可在手机上完成观察、发令、确认与资源投递闭环

## 15. 建议的 MVP 与后续边界

MVP 先做：

- Session 切换
- 输入发送
- 输出查看
- 两行底部输入区
- 终端模式
- 基础工具行
- 高危确认

MVP 后优先做：

- 对话模式
- FlowMind
- 图片/文件投喂
- Prompt 动作卡
- 输出聚合

后续增强再做：

- NAS 资源桥接
- 多工具词库联动
- 自动摘要与命令建议
- 更完整的自愈与告警体系

## 16. 最终建议

移动端不应追求“功能尽可能多”，而应追求“在手机上最快完成远程接管动作”。

因此产品上建议坚持以下顺序：

1. 先把 `Session 控制 + 输入 + 输出 + 确认` 做稳
2. 再把 `对话模式 + FlowMind + 快捷动作` 做顺
3. 最后再接 `资源桥接 + 文件投喂 + NAS`

如果后续进入实现阶段，建议再补一份“移动端控制台接口与事件协议文档”，把前端页面结构与服务端/桌面端消息字段进一步收敛为统一协议。
