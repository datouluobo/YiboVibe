## 2. 当前结论

当前仓库已经具备一套可编译、可打开页面、可发送文本、可回显输出的基础终端嵌入能力，但它仍属于“基础版终端页面”，还未达到 `specs-v2.md` 与 `architecture-vibe-coding.md` 所定义的完整 `Vibe Coding Console / WslAgentBridge / Headless Host` 目标。

当前已具备：

- 控制台页面路由与导航入口
- Tauri 侧终端启动、写入、关闭命令
- Windows ConPTY 基础桥接
- 前端基础终端/聊天双视图壳层
- Rust 与前端构建通过

当前主要缺口：

- 生命周期仍依赖页面挂载与卸载
- 只有单个全局终端，没有会话模型
- shell 启动能力仍为硬编码
- `stderr`、状态、尺寸同步链路不完整
- 还未接入输出解析、资产提取、远程控制等 v2 能力

## 4. 必须补齐事项

### 4.1 会话模型化

目标：

- 把当前“单个共享终端实例”升级为“可管理的终端会话”

执行清单：

- [x] 增加终端会话管理层
- [x] 为每个终端会话定义唯一标识
- [x] 支持创建、查询、切换、关闭会话
- [x] 为会话记录基础元数据
- [x] 明确会话当前状态

建议记录的会话信息：

- `session_id`
- `shell_kind`
- `cwd`
- `status`
- `started_at`
- `last_output_at`
- `exit_code`

完成标准：

- [x] 前端可基于 `session_id` 连接指定终端
- [x] 应用内不再只依赖一个全局终端实例
- [x] 会话状态可被查询并用于 UI 展示

### 4.2 页面与宿主解耦

目标：

- 让终端宿主不依赖页面生命周期存活

执行清单：

- [x] 页面卸载时不再默认销毁终端进程
- [x] 页面进入时优先附着到已有会话
- [x] 页面只负责订阅/取消订阅事件，不负责决定会话生死
- [ ] 明确“关闭页面”和“关闭会话”是两种不同动作

完成标准：

- [x] 切换页面后终端会话仍可继续存在
- [x] 重新进入页面后可恢复查看已有会话
- [x] 用户主动关闭会话前，宿主不会因页面切换被清理

### 4.3 补齐终端状态与退出信息

目标：

- 让上层页面能准确判断终端当前是否可用、是否退出、为何退出

执行清单：

- [x] 明确终端状态枚举
- [x] 补齐退出码记录
- [x] 补齐最后输出时间
- [ ] 区分“正在运行”“已退出”“启动失败”“已关闭”
- [x] 统一退出事件格式

完成标准：

- [x] UI 能展示当前会话状态
- [x] 会话退出后能看到退出结果
- [x] 状态变化可被上层逻辑消费

### 4.4 补齐 `stderr` 输出链路

目标：

- 把标准输出与错误输出分开管理

执行清单：

- [ ] 后端补齐错误输出事件 (Windows ConPTY 不分离 stderr，暂缓)
- [ ] 前端按输出类别分别渲染 (待后端支持)
- [ ] 错误输出与普通输出使用不同视觉层级
- [ ] 后续为错误提取、诊断和告警预留分类基础

完成标准：

- [ ] 前端监听到的 `stderr` 事件来自真实后端链路
- [ ] 错误输出不再混入普通输出流
- [ ] 错误输出可单独识别和展示

### 4.5 接通终端尺寸同步

目标：

- 让终端尺寸与实际展示区域保持同步

执行清单：

- [x] 为终端尺寸调整补齐命令入口
- [x] 前端感知终端容器尺寸变化
- [x] 建立前端尺寸变化到宿主 resize 的调用链
- [x] 保证窗口变化、面板变化后终端布局可更新

完成标准：

- [x] 调整窗口尺寸后终端显示正常
- [x] 行列变化能传递到宿主终端
- [x] 不再长期固定为单一默认尺寸

### 4.6 Shell 启动配置化

目标：

- 不再将终端宿主固定为单一 shell

执行清单：

- [ ] 支持多种 shell 类型
- [x] 支持默认 shell 配置
- [x] 支持为新会话指定启动宿主
- [ ] 预留工作目录与环境变量输入位置

建议首批支持：

- `cmd`
- `PowerShell`
- `WSL`

完成标准：

- [x] 新建会话时可明确宿主类型
- [x] 配置层可保存默认 shell
- [x] 不再依赖硬编码启动单一 shell

## 5. 体验优化事项

### 5.1 终端渲染层升级

目标：

- 把当前“文本流展示”升级为更接近真实终端的展示层

执行清单：

- [ ] 评估并接入真正的 terminal renderer
- [ ] 正确处理 ANSI 控制序列
- [ ] 支持交互式命令行基本显示行为
- [ ] 为后续全屏 CLI 或复杂交互工具预留基础

完成标准：

- [ ] 常见 CLI 的展示效果明显优于当前纯文本堆叠
- [ ] 光标、清屏、颜色等基础终端行为可被正确表现

### 5.2 输入能力补齐

目标：

- 让终端不只支持“输入一行文本并回车”

执行清单：

- [x] 补齐特殊键输入能力
- [x] 支持中断信号
- [x] 支持确认键、补全键、方向键
- [x] 区分“发送文本命令”和“发送控制动作”

优先支持：

- `Enter`
- `Ctrl+C`
- `Tab`
- `ArrowUp`
- `ArrowDown`
- `ArrowLeft`
- `ArrowRight`

完成标准：

- [x] 常见 CLI 的基本交互可完成
- [x] 用户可执行中断、确认和简单导航

### 5.3 终端视图与聊天视图职责拆分

目标：

- 明确终端流与消息流是两类不同内容

执行清单：

- [x] 不再把聊天视图仅作为终端内容换皮
- [x] 明确终端内容区、消息区、系统状态区的职责
- [x] 终端视图优先服务 shell 会话
- [x] 聊天视图优先服务会话消息与结构化动作

完成标准：

- [x] UI 层面对终端与聊天的边界清晰
- [x] 后续接入 Agent 消息流时无需推翻当前结构

### 5.4 会话恢复与会话列表

目标：

- 让用户能重新找到已有终端会话

执行清单：

- [x] 增加会话列表区域或切换入口
- [x] 支持显示当前活动会话
- [x] 支持重新附着到既有会话
- [x] 支持查看已退出会话的最后状态

完成标准：

- [x] 用户可以区分多个会话
- [x] 页面刷新或切换后可恢复已有会话上下文

### 5.5 诊断信息展示

目标：

- 让终端故障更容易定位

执行清单：

- [x] 展示当前 shell 类型
- [ ] 展示当前工作目录
- [x] 展示当前会话状态
- [x] 展示最后退出码
- [x] 展示启动失败或写入失败等错误提示

完成标准：

- [x] 常见终端异常可通过 UI 获得直接线索
- [x] 不必依赖日志才能判断最基本故障

## 7. 建议实施顺序

第一阶段：先补主链

- 会话模型化
- 页面与宿主解耦
- 状态、退出码、`stderr` 补齐
- `resize` 接通
- shell 配置化

第二阶段：补可用性

- 终端渲染层升级
- 输入能力补齐
- 会话恢复与会话列表
- 诊断信息展示
- 职责拆分

第三阶段：对齐 v2 主线

- `WslAgentBridge`
- 输出解析与提示识别
- Artifact Auto-Harvesting
- Vital Signs
- 远程控制协议
- Resource Bridge
