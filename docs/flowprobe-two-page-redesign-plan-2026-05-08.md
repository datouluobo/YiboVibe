# FlowProbe 双页面重构方案与执行计划

更新时间：2026-05-08

## 1. 目标

本方案用于将当前单页 `FlowProbe` 重构为两个独立页面，解决以下问题：

- 当前“测试”页同时承载 API 资产管理、本机转发、运行状态、统计和日志，信息架构失真
- 页面块过多，首屏留白大，主流程不清晰
- “测试”这个导航命名无法覆盖真实职责
- 接入管理和转出控制属于两类一级任务，不适合继续塞在同一路由里

本次重构的目标不是只调整视觉，而是先修正导航语义、页面骨架和工作流。

## 2. 结论

### 2.1 导航拆分

侧边栏直接将当前 `测试` 拆成两个一级入口，平铺显示：

- `接入`
- `转发`

页面标题统一为：

- `FlowProbe 接入`
- `FlowProbe 转发`

英文命名统一为：

- `Probe Sources`
- `Probe Relay`

说明：

- 侧边栏使用短中文名，保持与 `总览`、`灵思`、`键谱`、`规则`、`设置` 的现有风格一致
- 页面标题保留 `FlowProbe` 前缀，避免两个页面脱离产品语境

### 2.2 页面职责

`接入` 页负责“上游 API 资产管理”：

- API 列表
- 搜索 / 筛选 / 标签
- API 详情编辑
- 单个 API 连通性测试
- 模型拉取
- 设为 OpenAI / Anthropic 转发目标

`转发` 页负责“本机兼容接口运行与路由控制”：

- 运行状态
- Host / Port / Local Token
- OpenAI-compatible 当前目标
- Anthropic-compatible 当前目标
- IDE / 客户端接入地址
- 转发统计与近期活动

## 3. 接入页方案

## 3.1 核心 UX

`接入` 页采用固定的 `列表 -> 详情 -> 验证` 工作流。

布局建议：

- 左栏：API 列表
- 中栏：当前选中 API 的详情编辑
- 右栏：当前选中 API 的验证结果与模型列表

交互原则：

- 点击左侧 API 列表项后，中栏详情必须随选中项切换
- 右栏验证结果也必须切换到当前选中项
- 不弹窗编辑，不跳转到独立详情页
- 新增 API 时，直接在列表插入新对象并自动选中

这样可以保证整个页面始终围绕“当前选中的 API”展开，避免对象上下文丢失。

## 3.2 左栏：API 列表

左栏默认展示摘要信息：

- 名称
- 协议
- 服务商
- 状态
- 默认模型
- 标签

顶部控件：

- 搜索框
- 筛选芯片：`全部`、`OpenAI兼容`、`Anthropic`、`Ollama`、`本地`
- `新增 API`
- 可选 `导入`

列表项原则：

- 使用紧凑摘要，不展示长表单
- 选中态明确
- 协议类型和连通状态要在列表级别可扫读

## 3.3 中栏：API 详情

中栏只展示当前选中 API 的可编辑配置。

字段建议：

- 名称
- 服务商
- 协议
- Base URL
- API Key / Secret
- 默认模型
- 备注
- 标签
- 输入价格 / 1M
- 输出价格 / 1M

底部动作固定为：

- `保存`
- `测试 API`
- `拉取模型`
- `设为 OpenAI 目标`
- `设为 Anthropic 目标`

说明：

- “设为目标”是跨页联动动作，但仍应该放在接入页对象动作区
- 中栏不展示统计，不展示全局运行状态

## 3.4 右栏：验证结果与模型

右栏不是第二份详情，而是“当前 API 的验证面板”。

建议只保留：

- 连通状态
- 最近一次测试结果
- 响应时间
- 最近错误
- 可用模型列表

其中模型列表需要考虑“同名模型”问题。

### 3.4.1 同名模型处理

不同服务商、不同网关甚至同一家不同入口，可能返回同名模型。不能只用模型名作为唯一标识。

数据层要求：

- 模型唯一键至少使用 `source_id + model_name`

展示层要求：

- 模型名作为主标题
- 明确标注来源 API / 服务商
- 默认模型下拉框不能只显示裸模型名

交互原则：

- 在 `接入` 页中，模型列表只显示“当前 API 拉回来的模型”，不跨源混排
- 只有在转发目标选择或路由比较场景，才需要跨 API 观察

## 4. 转发页方案

## 4.1 核心原则

`转发` 页不应继续维持当前多块拼盘结构。页面应压缩成一个“运行控制台”，把当前分散的状态、路由、示例和统计收拢为固定骨架。

目标是从“7 块功能区域”收敛为 `3 个主区域`：

- 顶部整合状态条
- 中部两个转出口主卡
- 底部近期活动区

不再默认铺开大量“查看全部”的折叠列表。

## 4.2 顶部整合状态条

顶部状态条直接整合以下内容：

- 运行状态
- Host
- Port
- Local Token
- 保存配置
- 刷新
- 启动 / 停止
- 接入示例入口

说明：

- `运行状态`、`Host / Port / Token`、`接入示例` 可以放在同一套结构里，不必拆成独立卡片
- 接入示例不需要默认展开为大块说明，可采用内联复制区或轻量弹层

## 4.3 中部主区：两个转出口卡片

中部主区只保留两张主卡：

- `OpenAI 兼容接口`
- `Anthropic 兼容接口`

每张卡固定展示：

- 本地地址
- 当前目标 API
- 当前模型
- 最近请求
- 成功率
- Token
- 平均延迟

每张卡固定动作：

- `切换目标`
- `复制地址`
- `测试当前`

这两张卡可以与顶部状态条在视觉上保持连续关系，但仍应作为独立主对象存在，避免把所有字段压进单行造成可读性下降。

## 4.4 接入示例

接入示例不再作为独立大区域存在。

建议处理方式：

- 顶部状态条提供 `接入示例` 入口
- 默认只展示最关键的两条：
  - 本机回环地址
  - WSL / 局域网地址
- 完整示例通过抽屉、弹层或展开区查看

这样既保留实用性，也避免主页面再次被说明文本占满。

## 4.5 底部活动区

底部活动区只保留一个紧凑区域，用于展示近期运行情况。

建议内容：

- 最近请求
- 最近错误
- 最近切换记录

展现形式：

- 一张紧凑表格，或单列时间流
- 页面只展示最近若干条
- 完整历史通过单独入口查看

不建议在主页面上堆多组“查看全部”卡片。

## 5. 统计口径

## 5.1 默认展示口径

主页面默认展示两层统计：

- 全部转出汇总
- 按兼容接口拆分

也就是：

- 全局今日请求 / 成功 / 失败 / 输入 Token / 输出 Token / 平均延迟
- OpenAI-compatible 独立统计
- Anthropic-compatible 独立统计

## 5.2 明细口径

按目标 API 的统计应保留，但不建议默认摊开。

第三层统计维度为：

- 按当前目标 API 拆分

例如：

- OpenAI 当前转到 `Mimo` 的请求量 / Token / 成本
- Anthropic 当前转到 `New Target` 的请求量 / Token / 成本

这些明细更适合放在：

- 转出口卡片内的明细抽屉
- 底部活动区筛选结果
- 独立历史页

## 7. 执行计划

## 7.1 Phase 1：导航与路由拆分

目标：

- 将当前 `测试` 拆为 `接入` 和 `转发`
- 新建两个独立路由和页面骨架

任务：

- 更新 `App.tsx` 路由
- 更新 `Layout.tsx` 侧边栏导航
- 更新 `zh.json` / `en.json` 命名与 tooltip
- 为两个页面建立空壳布局和标题区

完成标准：

- 侧边栏出现 `接入`、`转发`
- 原 `测试` 入口下线
- 两个新页面可独立访问

## 7.2 Phase 2：接入页落地

目标：

- 从当前单页中拆出 API 资产管理工作流

任务：

- 提取 API 列表区
- 提取当前选中 API 的详情编辑区
- 提取当前 API 的测试结果与模型区
- 实现“点击列表项 -> 详情与验证结果同步切换”
- 实现“设为 OpenAI / Anthropic 目标”动作

完成标准：

- 接入页不再展示代理运行状态、日志总表、全局统计
- 新增 / 选择 / 编辑 / 测试 / 拉模型 / 设目标形成闭环

## 7.3 Phase 3：转发页落地

目标：

- 形成运行控制台视图，替代当前多块拼盘布局

任务：

- 整合顶部状态条
- 整合两张转出口主卡
- 收敛接入示例展示方式
- 将统计收敛为总览 + 按兼容接口拆分
- 将近期请求 / 错误 / 切换记录压成一个活动区

完成标准：

- 主页面不再出现多组零散折叠块
- 用户在单屏内能完成“看状态 -> 看当前路由 -> 复制接入地址 -> 测试当前”

## 7.4 Phase 4：统计与数据口径补齐

目标：

- 补足转发页需要的维度化统计

任务：

- 核对当前 `ProbeStatsPayload` 是否已支持按路由拆分
- 如不足，补充 Tauri / Rust 侧聚合结构
- 区分全局统计、按兼容接口统计、按目标 API 明细

完成标准：

- 转发页不依赖前端临时拼接推导
- 统计口径在接口定义中明确表达

## 7.5 Phase 5：回归与清理

目标：

- 清理旧实现，避免双轨页面长期并存

任务：

- 清理旧 `FlowProbe.tsx` 中已迁移区域
- 清理旧样式命名和废弃文案
- 更新相关说明文档

完成标准：

- 不再保留语义冲突的旧单页入口
- 代码结构与导航结构一致

## 8. 验收标准

验收时至少确认以下几点：

1. 侧边栏导航语义清晰：`接入` 管上游，`转发` 管下游。
2. `接入` 页点击 API 列表项后，详情和验证结果同步切换。
3. 同名模型不会因裸模型名冲突而混淆来源。
4. `转发` 页不再出现当前版本那种大面积留白和多块拼盘。
5. `转发` 页默认即可看见运行状态、当前目标、接入地址和基础统计。
6. 统计至少能区分全局汇总与按兼容接口拆分。

## 9. 风险与注意事项

- 不建议继续在现有 `FlowProbe.tsx` 上叠加 tabs。这样只会把当前信息架构问题藏起来，不会解决导航语义错误。
- 不建议先做视觉精修再做拆页。页面骨架和职责边界未定，视觉层优化会反复返工。
- 若后端统计当前只有全局汇总，转发页应先按现有能力落地，再补按接口 / 按目标 API 的维度化聚合，避免前端先造假口径。
- 接入页和转发页虽然拆开，但仍共享同一套 API 资产数据，不能复制出两套配置源。
