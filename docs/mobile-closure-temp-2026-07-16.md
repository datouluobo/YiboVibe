# 移动端收口清单（临时）

日期：2026-07-16

## 1. 核心定位

移动端不是全能远程开发工具，也不是脱离桌面端独立存在的开发环境。

移动端在 YiboVibe 中的职责应收束为：

- 用户暂时离开电脑时的低摩擦接管工具
- 在桌面端持续执行真实任务时，临时接手消息收发、快捷发令、命令确认与异常处理
- 帮助用户观察桌面端当前会话状态，并在需要时完成远程确认、拒绝、终止或恢复
- 用户回到电脑后，继续在原桌面上下文中无缝衔接，而不是在手机上形成独立工作流

因此，产品判断标准不是“手机上能不能独立完成更多事情”，而是：

- 能不能不中断电脑上的真实执行进程
- 能不能在离桌期间可靠接住关键消息和关键确认
- 能不能让用户回到电脑后不丢上下文、不分裂工作流

## 2. 本次收口结论

继续保留两条并行工作台：

- 终端控制台
- IDE / AI 工作台

它们是并行关系，不做二选一，也不强行合并为同一页面。

但两条工作台都应服从同一个移动端定位：

- 桌面执行
- 移动接管
- 回桌续接

不再把以下内容作为移动端主线能力推进：

- 复杂资源投喂
- 手机本地文件上传驱动工作流
- 手机远程浏览桌面资源
- 手机侧独立资源管理面板
- 以 FlowSync / FlowDrop / NAS 桥接为核心的移动端产品扩张

如果未来仍保留图片、附件、文件相关能力，应限制在：

- 桌面端或上游工具回流结果在移动端展示
- 桌面端主动触发的轻量确认场景

## 3. 收口优先级

### P0：必须先打通的主链路

#### 3.1 消息发送与回流稳定性

目标：

- 移动端在离桌期间能稳定发送消息
- 桌面端能稳定接收并继续执行
- 移动端能可靠看到回流状态，而不是只停留在“已发送/发送中”

移动端：

- 明确区分 `发送中`、`桌面已接收`、`桌面执行中`、`回流完成`、`发送失败`
- 优化 pending / queued / failed 状态的可见性
- 在弱网、断线、桌面未在线时给出明确状态提示

桌面端：

- 保持当前 `workbench snapshot` 回流主链路可用
- 对移动端发起的 turn/start 给出更稳定的接收与执行回执
- 减少“桌面已执行但移动端感知不到”的状态裂缝

服务端 / 协议：

- 继续沿现有 `workbench:snapshot` 与 `workbench:snapshot:status` 收口
- 保证消息状态字段足够稳定，便于移动端明确呈现

#### 3.2 远程确认 / 拒绝 / 终止

目标：

- 用户离桌时可以在手机上完成关键确认，不阻塞电脑进程

移动端：

- 现有审批卡补齐 `终止`
- 将等待确认状态更明显地挂到会话列表、会话头部和消息流中
- 为高危确认保留二次确认

桌面端：

- 补齐对 `确认 / 拒绝 / 终止` 的统一处理
- 对移动端发起的确认动作返回明确成功/失败状态

服务端 / 协议：

- 统一确认类消息结构
- 支持明确回执：已接收、已执行、已失败、已过期

#### 3.3 会话观测能力

目标：

- 用户离桌时能快速判断“电脑现在在干什么，是否需要我接手”

移动端：

- Sessions 侧边栏补齐关键观测信息：
  - 最后输出时间
  - 是否等待输入
  - 是否存在错误
  - 未读变化
  - 当前状态
- 终端控制台与 IDE 工作台共享同一套基础状态信号

桌面端：

- 在快照中补齐更稳定的 session summary 字段
- 尽量统一终端会话与 AI 会话的摘要模型

### P1：让移动端真正适合“离桌临时接手”

#### 3.4 告警、Vital Signs、自愈入口

目标：

- 不只是“能看到输出”，而是能及时发现异常并采取最小接管动作

移动端：

- 增加最近告警入口
- 增加 host / session 基础状态显示：
  - 当前状态
  - 心跳是否超时
  - 最后输出时间
  - 运行时长
- 提供最小自愈动作入口：
  - 重新拉快照
  - 重新请求状态
  - 重试连接
  - 远程终止卡死会话

桌面端：

- 稳定产出 `host:heartbeat`、`host:alert`、`host:vitals` 可消费内容
- 把 host 级信号和会话级信号区分开

服务端 / 协议：

- 明确 `host:*` 事件 payload
- 增加告警级别、来源、关联会话、推荐动作字段

#### 3.5 快捷动作与 FlowMind 收口

目标：

- 减少手机上输入长命令、长提示词的成本

移动端：

- 继续保留并强化快捷动作：
  - `Enter`
  - `Ctrl+C`
  - `y`
  - `n`
  - `重试`
  - `总结最近输出`
- `FlowMind` 不再只是硬编码命令模板，而是输入增强层
- 区分终端侧和 AI 工作台侧的输入增强内容

桌面端：

- 后续提供更结构化的模板、候选、最近动作、上下文摘要
- 让移动端少写硬编码，多消费桌面端整理后的上下文

#### 3.6 输出聚合与对话模式

目标：

- 让用户在手机上更快看懂“电脑最近发生了什么”

移动端：

- 强化消息流中的结果摘要、错误摘要、等待确认状态
- 对话模式继续推进，但不追求做成独立聊天 App
- 终端模式和对话模式都服务于“接手”而不是“替代桌面开发”

桌面端：

- 为移动端提供更可读的输出摘要和技术事件分类
- 尽量减少移动端自己二次猜测终端语义

### P2：结构整理与长期维护

#### 3.7 页面结构拆分

目标：

- 结束当前页面过大、过渡态长期存在的问题

移动端：

- 拆分 `AiWorkbenchPage`
- 将 session / tool / message / composer / config 等区域拆回独立组件
- 保持终端控制台和 IDE 工作台各自独立，但共享底层状态模型

#### 3.8 测试补齐

移动端：

- 增加 provider / widget 测试，覆盖：
  - 消息发送状态
  - 会话切换
  - 审批卡
  - 断线恢复
  - 告警与状态展示

桌面端：

- 增加快照字段和 broker 转发测试
- 增加确认动作和状态回执测试

## 4. 明确不作为当前主线的事项

以下事项不作为当前移动端收口主线：

- 手机本地主动选文件并投递到工作流
- 手机端复杂图片/附件发送系统
- 手机远程浏览桌面文件系统
- 围绕 FlowSync / FlowDrop / NAS 的移动端资源桥接扩张
- 把移动端做成完整独立开发工具

这些能力即使以后保留，也应服从同一原则：

- 桌面持续工作
- 手机只做轻量接管
- 回桌无缝续接

## 5. 推荐开发顺序

1. 先收口 `消息发送/回流状态`
2. 再收口 `远程确认/拒绝/终止`
3. 再补 `会话观测`
4. 再补 `告警 / vitals / 自愈入口`
5. 再优化 `快捷动作 / FlowMind / 输出聚合`
6. 最后做 `页面拆分与测试补齐`
