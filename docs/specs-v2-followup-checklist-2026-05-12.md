## 2. 范围定义

本文档覆盖：

- `specs-v2.md` 中除终端桥接主链之外的主线事项
- 桌面端、服务端、移动端三端职责重组
- 核心 Vibe 模块
- 原有 `Flow*` 模块的 v2 定位落地

本文档暂不覆盖：

- 纯视觉润色类任务
- 低优先级命名统一与品牌包装
- 与当前主线无关的历史残留整理

## 3. 核心主线任务

### 3.1 Headless Host 常驻化

目标：

- 让桌面端真正成为长期驻留、可观测、可恢复的执行宿主

执行清单：

- [ ] 明确宿主生命周期不依赖前台页面
- [ ] 建立后台常驻运行模型
- [ ] 支持会话级恢复与状态保留
- [ ] 为异常退出、自愈和重启预留统一入口
- [ ] 让 UI 成为宿主的观察与控制界面，而不是宿主本体

完成标准：

- [ ] 关闭或切换页面后宿主核心能力仍然存在
- [ ] 桌面端具备长期运行一组会话的结构基础
- [ ] 后续接入远程控制时不需要重做宿主生命周期

### 3.2 服务端 `Signal + Session + Memory Hub`

目标：

- 让服务端从“同步后端”升级为“信令与会话中枢”

执行清单：

- [ ] 明确服务端的会话中枢职责
- [ ] 建立桌面端与移动端之间的命令路由模型
- [ ] 支持会话元数据与状态同步
- [ ] 支持心跳与告警转发
- [ ] 支持加密资产索引、暂存与通知
- [ ] 预留知识沉淀与全局会话资产管理入口

完成标准：

- [ ] 服务端能承担会话路由与状态汇聚职责
- [ ] 服务端边界仍保持“不替代桌面端执行”
- [ ] 多端会话流转不再依赖点对点临时逻辑

### 3.3 移动端 `Remote Console`

目标：

- 让移动端成为真正的远程控制台，而不是只读查看页

执行清单：

- [ ] 支持多频道会话切换
- [ ] 支持消息流和状态流展示
- [ ] 支持快捷动作发令
- [ ] 支持远程确认、拒绝、终止
- [ ] 支持会话观测与告警接收
- [ ] 支持基础资源投喂入口

完成标准：

- [ ] 移动端可独立承担远程接管入口角色
- [ ] 用户离开桌面后仍可完成最常见的控制动作

### 3.4 Multi-Agent Sessions

目标：

- 让系统支持多个独立并行会话，而不是单线串行控制

执行清单：

- [ ] 定义会话实体与会话元数据
- [ ] 支持多个独立会话并行存在
- [ ] 支持频道式切换
- [ ] 支持会话级启停与诊断
- [ ] 支持跨会话转发上下文与资产

完成标准：

- [ ] 每个会话拥有独立状态与控制面
- [ ] 多会话不会互相串流或污染状态

## 4. 五大 Vibe 模块任务

### 4.1 Context Projection

目标：

- 把桌面端当前开发上下文压缩为远程可消费的结构化快照

执行清单：

- [ ] 明确上下文快照的输入源
- [ ] 定义上下文快照的结构化输出
- [ ] 支持捕获当前项目路径、活动文件、末尾片段
- [ ] 支持汇总最近终端输出与错误
- [ ] 支持附带当前 Agent 状态
- [ ] 为远程唤醒和继续发令提供直接输入载荷

完成标准：

- [ ] 远程端能看到足够短但足够完整的上下文包
- [ ] 会话恢复和远程接管不再只依赖纯文本聊天记录

### 4.2 State-Aware Action Bar

目标：

- 降低移动端处理交互式终端确认的成本

执行清单：

- [ ] 识别典型交互提示
- [ ] 为提示生成结构化动作
- [ ] 支持一键确认、拒绝、终止
- [ ] 支持需要时的自定义输入槽
- [ ] 建立动作卡与终端控制动作之间的映射

完成标准：

- [ ] 常见确认型交互可以通过动作卡快速完成
- [ ] 用户不必频繁手打原始终端输入

### 4.3 Resource Bridge

目标：

- 把本地、NAS、附件类私有资源安全送达目标会话

执行清单：

- [ ] 明确资源桥接输入源
- [ ] 支持本地文件资源选择
- [ ] 支持 NAS / 私有文件入口
- [ ] 支持附件与结构化说明进入会话
- [ ] 明确桌面端负责真实读取、移动端负责发起意图

完成标准：

- [ ] 资源进入会话有统一入口
- [ ] 多端资源投喂不再依赖临时复制粘贴流程

### 4.4 Agent Vital Signs

目标：

- 让后台 Agent / 会话具备可观测性

执行清单：

- [ ] 定义基础运行指标
- [ ] 支持 CPU、内存、运行时长等信号
- [ ] 支持最后输出时间与当前状态
- [ ] 支持心跳超时和静默检测
- [ ] 支持桌面端与移动端统一消费

完成标准：

- [ ] 用户可以区分“正在运行”“静默等待”“可能卡死”
- [ ] 告警与自愈逻辑具备可依赖的信号源

### 4.5 Artifact Auto-Harvesting

目标：

- 自动从终端与消息流中提取可复用资产

执行清单：

- [ ] 支持提取代码块
- [ ] 支持提取 patch
- [ ] 支持提取 shell 命令
- [ ] 支持提取错误摘要
- [ ] 支持提取路径和文件引用
- [ ] 明确资产后续流向

完成标准：

- [ ] 高价值输出可被结构化复用
- [ ] 后续复制、同步、转发、沉淀具备统一资产入口

## 9. 建议实施顺序

第一阶段：宿主与中枢基础

- Headless Host 常驻化
- 服务端 `Signal + Session + Memory Hub`
- Multi-Agent Sessions
- FlowDeck 总控台定位重建
- Settings 配置中心扩展

第二阶段：远程可用性

- 移动端 `Remote Console`
- Context Projection
- State-Aware Action Bar
- Agent Vital Signs
- 高危控制与二次确认

第三阶段：资源与资产流转

- Resource Bridge
- FlowDrop 资源投递能力
- FlowSync 资产同步底座
- Artifact Auto-Harvesting
- 跨会话资产流转

第四阶段：模块协同与沉淀

- FlowSnap 模板层
- FlowHint 候选增强层
- FlowRules 会话策略层
- FlowKeys 远程操作映射层
- 知识沉淀
- 信息架构最终收束
