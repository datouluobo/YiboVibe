# YiboVibe v2 Console UI 规范

更新时间：2026-05-11

## 1. 文档目的

本文档定义 YiboVibe v2 管理控制台与 Vibe Coding 控制台的统一 UI 规范。

目标不是做视觉风格展示，而是给桌面端后续页面提供稳定的布局、信息密度、交互和状态表达约束，尤其适用于：

- `Console`
- `Agents`
- `Sessions`
- `Resources`
- `FlowProbe`
- 管理后台类页面

## 2. 设计方向

v2 控制台采用：

- `Modern Enterprise`
- `High Density`
- `Low Friction`
- `Operator First`

关键词：

- 高密度但不混乱
- 强状态表达
- 局部滚动替代全局滚动
- 一屏内完成监控、操作和验证闭环

## 3. 全局布局原则

### 3.1 Zero Global Scroll

控制台页面默认遵循 `Zero Global Scroll`：

- 页面使用固定视口高度
- 外层容器不出现整页纵向滚动
- 超出内容仅在局部面板内部滚动

适用页面：

- `FlowProbe`
- `Sessions`
- `Console`
- 各类多面板管理页

### 3.2 固定分区

推荐布局：

- 顶部：标题区 + 快捷动作区
- 中部：核心控制区
- 底部或右侧：日志 / 结果 / 诊断区

对于双区页面，推荐比例：

- `4:6`
- `5:5`
- `3:7`

根据内容选择，但比例要稳定，避免页面高度抖动。

### 3.3 局部滚动

以下区域允许滚动：

- 日志流
- 资源列表
- 模型标签池
- 会话消息流
- 诊断结果区

以下区域默认不滚动：

- 页面标题
- 主操作栏
- 核心配置摘要

## 4. 信息架构原则

### 4.1 一屏闭环

单个控制台页面应尽量在一屏完成：

- 输入
- 状态
- 验证
- 结果
- 下一步动作

避免把“配置在一个页、诊断在另一个页、结果在第三个页”拆散。

### 4.2 主次分层

页面信息分为三层：

- 一级：当前任务必须看的状态和动作
- 二级：诊断、统计、上下文
- 三级：低频配置和高级选项

三级信息默认收纳，而不是默认展开。

### 4.3 稳定位置

关键元素位置要稳定：

- 主按钮位置稳定
- 健康状态点位置稳定
- 搜索框位置稳定
- 结果面板位置稳定

不要因为状态切换导致布局跳跃。

## 5. 表单与配置面板规范

### 5.1 双列紧凑表单

配置表单优先使用双列 Grid：

- 左右对齐
- 纵向节奏紧凑
- 标签、输入框、状态提示统一基线

适用于：

- FlowProbe 接入配置
- 转发规则
- Agent 启动参数
- 资源桥接参数

### 5.2 字段优先级

高频字段默认展示：

- 名称
- Base URL / Host
- 认证字段
- 默认模型
- 启停开关
- 主要路径

低频字段默认折叠：

- 计价单位
- 额外 header
- 高级协议选项
- 诊断阈值
- 兼容性开关

### 5.3 敏感字段

敏感字段规范：

- 默认掩码显示
- 支持一键显隐
- 支持一键复制
- 显示最近更新时间或保存状态

不要让敏感字段的展开/收起影响整行高度剧烈变化。

## 6. 状态表达规范

### 6.1 健康状态点

核心列表项必须支持高对比度状态点。

推荐颜色语义：

- 绿色：正常
- 红色：失败/断开
- 黄色：警告/部分异常
- 灰色：未知/未检测

状态点可带轻微呼吸动画，但不能夸张。

### 6.2 状态文案

状态文案应短且可扫描，例如：

- `Healthy`
- `Auth Failed`
- `Timeout`
- `Syncing`
- `Waiting Input`
- `Idle`
- `Running`

### 6.3 毫秒级延迟与基础统计

对于 API / 会话 / 代理列表项，状态区应能容纳：

- 平均延迟
- 最近成功时间
- 错误计数
- 简短协议标签

例如：

- `12ms`
- `Last ok 2m ago`
- `OpenAI`

## 7. 诊断与验证区规范

### 7.1 双栏验证面板

验证类页面推荐使用左右双栏：

- 左侧：诊断日志流
- 右侧：结构化结果或对象池

`FlowProbe` 推荐形式：

- 左半部：Request/Response/错误堆栈
- 右半部：模型池、协议结果、统计摘要

### 7.2 日志流

日志流规范：

- 等宽字体
- 明确区分 info/warn/error
- 自动滚动可暂停
- 支持复制单条
- 支持时间戳

错误要直指原因，例如：

- `401 Unauthorized`
- `DNS resolve failed`
- `Connection refused`

### 7.3 结构化结果区

结构化结果区优先展示：

- 协议识别
- 模型数量
- 认证状态
- 平均延迟
- 费用/Token 统计

用摘要卡片或标签，不要堆成长段文字。

## 8. 模型池与资源池规范

### 8.1 标签化展示

模型、资源、会话标签优先用高可读圆角标签。

要求：

- 颜色有层次但不过度花哨
- 支持 hover 详情
- 支持选中和多选
- 支持快速复制名称

### 8.2 搜索优先

对于可能出现大量项目的面板，顶部必须有极简搜索框。

适用：

- 模型池
- 资源池
- 会话列表
- 日志过滤

### 8.3 空状态

空状态要简短、明确、可执行。

例如：

- `No models loaded yet`
- `Start a session to see artifacts`
- `Select a resource source first`

## 9. 会话与消息流规范

### 9.1 多频道结构

多 Agent 会话界面应具备类似频道工作区的结构：

- 左侧会话列表
- 中部消息流
- 右侧上下文/资源/状态

### 9.2 消息气泡

消息流需要明确区分：

- 用户意图
- Agent 输出
- 系统状态
- 诊断事件
- 可执行动作卡

动作卡需要比普通消息更高优先级。

### 9.3 Action Bar

当系统识别出等待输入状态时，消息区附近需要突出显示动作卡：

- `YES`
- `NO`
- `ENTER`
- `ABORT`
- `CUSTOM INPUT`

动作卡必须：

- 高可见
- 可单手点击
- 不与普通消息样式混淆

## 10. 视觉节奏与交互细节

### 10.1 节奏

推荐节奏：

- 卡片间距紧凑
- 模块边界明确
- 重要状态用色块或边框强调
- 次要说明文字用较低对比度

### 10.2 动效

仅使用少量有意义的动效：

- 状态点呼吸
- 新日志进入
- 面板切换
- 会话状态变化

不要对所有控件统一加动画。

### 10.3 防抖动

要避免：

- 状态切换导致按钮横跳
- 折叠展开导致整页高度暴涨
- 加载态和完成态宽度变化过大

如果状态会切换，优先预留稳定宽度。

## 11. 桌面端控制台页面建议

### 11.1 Console

包含：

- 活跃会话摘要
- Vital Signs
- 最近异常
- 快捷控制
- 资源投喂入口

### 11.2 Agents

包含：

- Agent 模板列表
- 启动入口
- 环境信息
- 上游模型绑定

### 11.3 Sessions

包含：

- 会话频道列表
- 气泡流
- Action Bar
- Artifact 侧栏
- Resource 侧栏

### 11.4 FlowProbe

包含：

- 接入配置
- 转发配置
- 健康状态
- 日志诊断
- 模型池

## 12. 实现约束

后续实现时应遵循：

- 页面外层默认不出现全局滚动
- 组件尽量复用状态展示语义
- 列表、状态点、摘要卡、日志流保持统一视觉语言
- 管理台优先解决可操作性，而不是装饰性
